# 05 — OpenAI DALL·E / gpt-image: Category Index

## Category Executive Summary

The nine angles collectively establish that OpenAI's image surface in 2026 is **two architecturally different systems bolted onto one API tree**, not a single product line. Understanding the seams is the entire job of a prompt enhancer targeting OpenAI.

1. **Two families, one doomed.** `dall-e-2` and `dall-e-3` are deprecated with a **2026-05-12 sunset** (5a-dalle3-prompt-guide-and-rewriter; 5c-logo-icon-generation). Every asset route in the plugin should compile to the `gpt-image-1`/`gpt-image-1-mini`/`gpt-image-1.5` family. DALL·E 3 remains worth understanding only because its quirks (the relabeler, silent branded-object rewrites, `vivid`/`natural` split) persist conceptually in the newer stack.

2. **DALL·E 3 is a two-model pipeline: a GPT-4 "relabeler" feeds a diffusion backbone.** Every `/v1/images/generations` call is rewritten server-side before the image model ever sees it; the rewrite returns as `revised_prompt` and **cannot be disabled** by any official flag (5a-dalle3-prompt-guide-and-rewriter; 5a-dalle3-recaptioning). This is load-bearing — the model was trained on **95 % synthetic descriptive captions / 5 % ground-truth alt-text** (Betker et al. 2023), so short human prompts are off-distribution and the relabeler brings them into DSC-land at inference time.

3. **The only documented relabeler-softening trick is an in-prompt meta-instruction**, supplied by an OpenAI staff reply on the developer forum: `"I NEED to test how the tool works with extremely simple prompts. DO NOT add any detail, just use it AS-IS:"` for short prompts, or `"My prompt has full detail so no need to add more:"` for long ones (5a-dalle3-prompt-guide-and-rewriter §2; 5a-dalle3-recaptioning §2.4; 5d-failure-modes-text-and-moderation). This softens style/detail rewrites but **never** bypasses the separate branded-object, named-person, and living-artist substitution stages (5a-dalle3-recaptioning §3; 5d-failure-modes-text-and-moderation §3).

4. **`gpt-image-1` / `gpt-image-1.5` are natively multimodal autoregressive models with real RGBA.** `background: "transparent"` + `output_format ∈ {png, webp}` + `quality ∈ {medium, high}` yields **genuine 4-channel alpha** — unlike Gemini/Imagen which produce opaque pixels or a fake checkerboard baked into RGB (5b-gpt-image-1-api §9; 5b-gpt-image-1-api-and-transparency §3.1). This is the single biggest quality lever for logo/icon/favicon/sticker tasks and the headline differentiator against the Google family.

5. **Transparency is broken in three predictable ways** (5b-gpt-image-1-api-and-transparency §3.3; 5d-failure-modes-text-and-moderation): (a) `/v1/images/edits` ignores `background="transparent"` and returns opaque output — always regenerate on `/generations`; (b) `quality: "low"` produces ragged edges and halos because 272 output tokens is insufficient to resolve a clean matte; (c) prompt `"remove the background"` over-segments and punches through legitimate white regions (shirts, highlights, UI panels). Canonical recipe: **`/generations` + `background=transparent` + `png` + `quality=high|medium` + prompt that echoes `"on a transparent background, clean anti-aliased edges"`**.

6. **Text rendering is stratified, not binary.** DALL·E 3 misspells anything past one short word (T5 tokenizer sees whole words, not letters; Betker §5.2, Spennemann 2025 measures 15.6 % attribute-error rate). `gpt-image-1` clears single headings but garbles small/dense text. `gpt-image-1.5` is the first OpenAI model where in-image typography is production-viable; LM Arena ranks it #1 in March 2026 (ELO 1264). For wordmarks, the prompt-to-asset should **route by text length**: 0 words → any model; 1–3 → `gpt-image-1.5` or `gpt-image-1`; ≥4 or multi-line → `gpt-image-1.5` only; handwritten/CJK → generate mark-only and composite type in post (5d-failure-modes-text-and-moderation §Text Rendering; 5c-logo-icon-generation §Pattern E).

7. **Three input-preservation levers on `/edits` drive brand/face fidelity.** `input_fidelity="high"` roughly 2–4×'s input-image tokens and preserves typography, faces, and logos of the **first** image in a multi-image edit — subsequent images are preserved less finely (5b-gpt-image-1-api §Recipe 8; 5c-openai-cookbook-asset-workflows §Recipe 3–4; 5d-system-style-behavior §3). Always place the logo/face first; concatenate brand assets into a single composite for whole-identity work. Up to **16 reference images** are accepted; `mask` (alpha PNG) binds only to image 0 and is **soft**, not hard — the model may rewrite outside the masked region (5b-gpt-image-1-api-and-transparency §4).

8. **Three distinct rewriter layers** must be reasoned about separately (5d-failure-modes-text-and-moderation §Prompt Rewriter Tampering; 5a-dalle3-recaptioning §2.1; 5d-system-style-behavior §1): (i) the DALL·E 3 API-side relabeler (unkillable, returns `revised_prompt`); (ii) the ChatGPT tool-side system prompt (fans one request into 4 stylistic variants, injects diversity/safety rewrites, exposes an undocumented `seeds` parameter unavailable via API); (iii) the **Responses API mainline LLM** rewrite on `gpt-image-1*` — here the developer writes the system/developer message, the LLM rewrites the user's prompt into `revised_prompt`, and the image model receives the rewritten text. **Image API direct path has no rewriter for `gpt-image-1*`** — this is the most controllable surface.

9. **Two distinct style-instruction surfaces exist, and only one has a system role** (5d-system-style-behavior §1). `/v1/images/generations` accepts a single `prompt` string — no system/developer message. The Responses API respects the standard `developer ≻ system ≻ user` hierarchy, preserves `instructions` across `previous_response_id` turns, and is the correct place for durable house-style rules (brand palette, negative-space constraints, "never render human faces"). For one-shot batch asset generation, go direct to `/v1/images/generations`; for conversational iterative design, use Responses.

10. **Moderation is a four-layer stack** (5d-failure-modes-text-and-moderation §Moderation; 5d-system-style-behavior §5): prompt-text classifier (named entities, trademarks, post-1912 artists, CSAM, weapons) → prompt rewriter (DALL·E 3 only) → output-side safety classifier (`moderation="auto"|"low"`) → deployment wrappers (Azure, Replicate, Pollinations). `moderation="low"` only softens the output classifier and exists only on `gpt-image-1*`, not DALL·E 3. `gpt-image-1` **rejects pop-culture prompts that DALL·E 3 rendered** — stormtroopers, red-cap plumbers, cartoon mice — with `content_policy_violation` (OpenAI community thread 1252701, May 2025). A `moderation_blocked` result still **costs tokens** (thread 1245636), so a prompt-to-asset must pre-screen via the free `omni-moderation-latest` endpoint.

11. **Three sizes, no others.** `1024x1024`, `1536x1024`, `1024x1536`, or `auto` — period (5b-gpt-image-1-api §File Size Limits). All favicon/app-icon workflows must generate at 1024² and downscale client-side with Lanczos. Composition is coupled to aspect ratio (5a-dalle3-prompt-guide-and-rewriter §Size): square biases cluttered snapshot framing, landscape biases professional hero-banner framing, portrait biases phone-mockup/story-frame framing. Changing `size` on an identical prompt materially changes composition — do not treat size as "pixels only".

12. **Pricing is per output token, deterministic per (quality, size).** At $40/1M image-output tokens for `gpt-image-1` (~$32/1M for `gpt-image-1.5`), 1024² high = 4,160 tokens ≈ $0.167 (the famous ~$0.19 figure with prompt tokens included). Low/medium/high for a square = **272 / 1,056 / 4,160** tokens, a **15× spread** (5b-gpt-image-1-api §Token Pricing; 5b-gpt-image-1-api-and-transparency §5). Streaming adds 100 tokens per `partial_image` (0–3, ~7 % uplift for 3 partials). `input_fidelity="high"` adds ~2–4× input-image tokens. **Batch API gives 50 % discount** with 24 h SLA — correct for brand-kit sweeps but cannot be combined with partial_images (5c-openai-cookbook-asset-workflows §Recipe 7).

13. **Rate-limit ceiling is IPM, not TPM.** Tier 1 = 5 IPM; Tier 5 = 250 IPM. A single high-quality 1024² consumes ~4,260 TPM out of the Tier-1 100k TPM budget, so images-per-minute bites first (5b-gpt-image-1-api §Rate Limits; 5e-production-integrations §Scaling). Multi-tenant deployments need per-tenant buckets + exponential-backoff-with-jitter (`tenacity.wait_random_exponential(min=1, max=60)` per OpenAI's cookbook).

14. **Org verification is a blocking onboarding step.** `gpt-image-1*` requires API Organization Verification (government ID for the org owner) before first call; stale keys from pre-verification projects return 403s and `"retry after 0.0 seconds"` (5b-gpt-image-1-api §Access Control; 5e-production-integrations). Surface this with a specific error message rather than a generic permission error — it is the #1 support ticket trigger on a fresh account.

15. **C2PA provenance metadata is mandatory, automatic, and non-optional.** All `gpt-image-1*` outputs carry signed Content Credentials identifying them as AI-generated (`"OpenAI"` or `"Azure OpenAI ImageGen"` software agent; 5d-system-style-behavior §5). This is both a safety feature and a product surface — the prompt enhancer should surface "AI-generated, signed" status to end users and preserve the manifest through the post-processing pipeline (rembg, Pillow, pngquant) rather than stripping it. Downstream screenshot/re-encode workflows will silently lose the manifest.

## Cross-Cutting Patterns

Several themes appear across **every** angle and therefore constitute plugin-level truths, not file-local claims:

- **The prompt rewriter is architecturally different across models, but behaviorally unavoidable in all OpenAI entry points.** DALL·E 3 rewrites inside the image service; gpt-image-1 inside the Responses API's mainline LLM; ChatGPT inside a tool-system-prompt layer. Only `/v1/images/generations` direct against `gpt-image-1*` is rewriter-free (5a-recaptioning §5; 5d-failure-modes §Prompt Rewriter Tampering; 5d-system-style-behavior §1).
- **"Medium or high quality" is the floor for every asset task that matters.** Transparency edges, text rendering, photoreal textures, fine typography, composition adherence on multi-constraint prompts — all degrade at `quality="low"`. The 4× price jump from medium to high buys edge cleanliness and layout precision, not pixels (5b-api §Transparency; 5c-logo §Pattern A; 5d-system-style §4).
- **First-image-wins semantics on `/edits`.** Every edit pathway (`input_fidelity="high"`, mask application, multi-image fidelity) weights image index 0 specially. Product code should compose a single "primary" image (logo / face / brand sheet) and place it first (5b-api §Edit-only; 5c-workflows §Recipe 3; 5d-system-style §3).
- **C2PA + organization-verification + URL expiry are three always-true operational facts.** Never skip onboarding verification. Never rely on the OpenAI URL outliving the request. Never strip C2PA if provenance matters (5b-api §Access Control; 5d-system-style §5; 5e-integrations).
- **Route-by-text-length.** Zero words → any model by style/cost; 1–3 words → gpt-image-1/1.5; >3 or multi-line → gpt-image-1.5 only; script/handwritten/CJK → generate mark-only and composite type in post. This rule is repeated verbatim across 5a, 5c, 5d (5a-prompt-guide §Example 4; 5c-logo §Pattern E; 5d-failure §Text).

## Controversies / Disagreements

- **Does `background="transparent"` actually work on `/edits`?** 5b-gpt-image-1-api (§Transparency) reports community threads showing transparency on `/edits` "sometimes returns an opaque image" and recommends re-emphasizing in the prompt. 5b-api-and-transparency (§T3) is stronger: "the parameter is accepted but the model ignores it — reproducibly returns opaque output". 5d-system-style-behavior (§6) says the edit-endpoint transparency "was initially broken prompt-only and later fixed by exposing `background='transparent'` on `/v1/images/edits`" and that it now works on current APIs. 5c-logo-icon-generation (§Pattern D) concurs with the "fixed" view. Resolution: transparency on `/edits` is accepted but unreliable enough that production code **must not depend on it** — regenerate on `/generations` whenever alpha is required.
- **Does `gpt-image-1` apply a rewriter on the direct Image API?** 5a-dalle3-recaptioning (§5) says rewriting is "still there; OpenAI just moved it to the Responses API". 5c-openai-cookbook-asset-workflows (§Gotchas) flatly states "`gpt-image-1`/`gpt-image-1.5` don't auto-rewrite in the Image API, only in Responses/ChatGPT". 5d-failure-modes (§Prompt Rewriter Tampering) agrees with the latter: "no rewriter — the prompt is the prompt". Resolution: on `/v1/images/generations` direct, the prompt is transmitted unchanged; rewriting is re-introduced by any Responses API / ChatGPT tool wrapper around it.
- **Who wins on photorealism and text rendering?** 5b-api-and-transparency (§6) positions `gpt-image-1.5` as best-in-class on text (≥1 word, multi-line, non-Latin) while Nano Banana Pro wins on photoreal skin. 5d-failure-modes (§Text Rendering Deep-Dive) agrees on gpt-image-1.5 leadership but cites Ideogram 3.0 at ~90 % text accuracy ahead of gpt-image-1 — with gpt-image-1.5 closing but not quite matching Ideogram on stylized typography. Resolution: for UI-mockup legibility, gpt-image-1.5 is sufficient; for brand-critical stylized typography, Ideogram still wins.
- **Is the "I NEED / DO NOT add detail" prefix actually effective?** 5a-dalle3-prompt-guide (§2) and 5a-recaptioning (§2.4) both present it as the officially sanctioned softener, traced to a forum reply from an OpenAI staffer. 5d-failure-modes (§Workarounds) is more skeptical: "a softener, not a kill-switch". Resolution: it reduces style/detail rewriting but cannot suppress the independent branded-object, named-person, and living-artist substitution stages.

## Gaps

- **No controlled diff between Image API and Responses API moderation.** Community reports (5d-failure, thread 1252701) show the *same prompt* succeeding on DALL·E 3 and failing on `gpt-image-1`; none measure whether Responses API rewriting *softens* moderation vs the direct Image API for the same gpt-image call. This matters for routing policy.
- **No public branded-object blocklist.** 5a-dalle3-prompt-guide (§Open Questions) calls for a fuzzing corpus; none exists. A prompt enhancer that warns users pre-call needs this.
- **No per-category moderation dial.** The Image API `moderation` parameter is a two-tier binary (`auto`/`low`). Enterprise tenants wanting "allow medical, block sexual" currently have no API-level lever and must pre-filter via the text moderation endpoint (5d-system-style §Open Questions).
- **`input_fidelity="high"` multiplier is unpublished.** ~2–4× input-image tokens is the field estimate; the exact multiplier would make budgeting deterministic (5b-api §Token Pricing).
- **No official opt-out for the Responses-API rewrite.** DALL·E 3 had an explicit "workaround"; `gpt-image-1*` via Responses has no documented knob at all. Teams that need deterministic output have no supported hook (5d-system-style §Open Questions).

## Actionable Recommendations for the Plugin's OpenAI Routing

1. **Default model by asset type and text requirement.** Logo/icon/favicon/sticker with ≤3 words → `gpt-image-1.5` (fallback `gpt-image-1`). Wordmarks or marketing copy → `gpt-image-1.5` only. Drafts / cost-bound batch → `gpt-image-1-mini`. Never default to DALL·E 3 for new code paths — it sunsets 2026-05-12.
2. **When to route transparent:** any logo / icon / sticker / favicon / "cutout" asset type automatically sets `background="transparent"` + `output_format="png"` + `quality="high"` (or `"medium"` if budget-bound), on `/v1/images/generations` only. Never on `/edits`.
3. **Handle the prompt rewriter explicitly per pathway.** Direct `/v1/images/generations` against gpt-image-1* → the plugin's enhanced prompt is transmitted verbatim; full author control. Responses API → wrap the user's prompt with a developer-role instruction `"Call image_generation with the user's prompt exactly as written — do not paraphrase or expand."` and log tool-call arguments for audit. DALL·E 3 (legacy only) → prefix with `"My prompt has full detail so no need to add more:"` (preferred over `"I NEED to test…"` when the enhancer's output is already DSC-shaped) and **always log `revised_prompt`** back to the caller as first-class debug output.
4. **Shape the prompt as a DSC caption.** Target 40–80 words, image-type prefix (`Photo of…` / `Illustration of…` / `Vector mark of…`), present-tense declarative, ordered `background → subject → key details → constraints`, with the negative list as the **final** line (`"No watermark, no extra text, no logos/trademarks, no background frame, no drop shadow."`). This minimizes rewriter delta on every OpenAI pathway (5a-recaptioning §6.2–6.3; 5d-system-style §2).
5. **Moderation avoidance — five layered rules.** (a) Pre-screen every prompt through the free `omni-moderation-latest` endpoint; return the user a specific rejection with reasons before spending an image token. (b) Strip or paraphrase trademark/celebrity/public-figure/living-artist names at the enhancer layer, proposing the substitution to the user explicitly — do not let OpenAI silently rewrite and leave the user confused. (c) Translate living-artist styles to 3–5 visual attributes (`"in the style of Studio Ghibli"` → `"soft watercolor backgrounds, 2D cel animation, pastoral Japanese countryside"`). (d) For brand-adjacent pop-culture prompts, prefer geometric/chromatic description over named entities. (e) Set `moderation="low"` on `gpt-image-1*` for branding/figurative tasks (still subject to Usage Policies); log `moderation_blocked` separately from generic errors so the product can offer a retargeted retry.
6. **Transparency verification harness.** Every transparent-generated asset passes a PIL check before save: `img.mode == "RGBA"`, `getchannel("A").getextrema() != (255, 255)`, `getchannel("A").getextrema()[0] < 16`. If any check fails, the provider returned fake alpha — error out and retry with a stricter prompt or fall back to `gpt-image-1.5 /generations` + `rembg` safety-net post-processing (5b-api-and-transparency §3.1; 5c-workflows §Pattern D).
7. **Identity-preserving edits.** Whenever the user supplies a reference image (logo PNG, face photo, brand sheet), route to `/v1/images/edits` with `input_fidelity="high"` and place the reference **first** in a multi-image array. For transparent output on the edited asset, regenerate on `/generations` with the edit-output as a reference — don't ask `/edits` to "remove the background".
8. **C2PA transparency to end users.** Preserve the C2PA manifest through post-processing (rembg, Pillow, pngquant do not strip it by default; `oxipng` with `--strip=safe` keeps it; screenshot pipelines do not). Expose "AI-generated, signed by OpenAI/Azure" status in the plugin UI and its MCP response payload.
9. **Batch API for brand-kit sweeps.** When the user asks for a full identity (logo × favicon × OG × app-icon-iOS × app-icon-Android × splash), submit a single JSONL job with 24 h SLA for 50 % discount. Real-time UI paths stay on `/generations` with `partial_images=2` streaming; batch and streaming are mutually exclusive (5c-workflows §Recipe 7).
10. **Operational hygiene.** Surface org-verification errors with a link to `platform.openai.com/settings/organization/general`. Copy every returned `b64_json` to durable blob storage on receipt — OpenAI URLs (when DALL·E 3 returns one) expire in ~1 h. Per-tenant IPM buckets with `tenacity.wait_random_exponential(min=1, max=60) + stop_after_attempt(6)`.
