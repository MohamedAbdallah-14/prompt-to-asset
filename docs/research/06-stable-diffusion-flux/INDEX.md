# 06 — Stable Diffusion & Flux: Category Index

## Category Executive Summary

Across five angles (6a SD 1.5/2.1/XL, 6b Flux family, 6c conditioning adapters, 6d LoRA training, 6e local pipelines), open-weights image generation in 2024–2026 has bifurcated into two architecturally different stacks with very different prompting contracts, and a prompt-to-asset targeting software assets must handle both. The fifteen load-bearing insights:

1. **Stable Diffusion prompting is cross-attention manipulation, not English parsing.** Every A1111 DSL feature — `(word:1.2)`, `[from:to:when]`, `AND`, `BREAK` — is a direct operation on the 77-token CLIP tensor fed into UNet cross-attention [6a]. Porting prompts between UIs (A1111, ComfyUI, InvokeAI, Forge) fails because of weight-normalization differences, not syntax differences; `BlenderNeko/ComfyUI_ADV_CLIP_emb` with `token_normalization=mean` is the portability fix [6a].
2. **Flux rejects `negative_prompt` by design, not by oversight.** Flux.1 [dev/schnell/pro], Flux.1 Tools, Kontext, and Flux.2 are all guidance-distilled — CFG is baked into the student weights as a conditioning scalar, so there is no unconditional branch for a negative prompt to hijack [6b]. BFL's Flux.2 prompting guide makes this explicit; any enhancer that silently passes a `negative_prompt` to a Flux endpoint will either 400 or drop it [6b].
3. **Long, prose prompts are how Flux actually outperforms SDXL.** T5-XXL carries 512 tokens on Flux.1 and Mistral-3 24B VLM carries ~32K-effective on Flux.2 vs. CLIP's 77; BFL's Subject+Action+Style+Context scaffold plus JSON prompts with hex-coded subjects (`"Apple: #0047AB"`) are the structural replacements for tag-soup + negative-prompt hygiene [6b].
4. **SD wins when flat-vector / duotone / illustrated aesthetics matter and VRAM is tight; Flux wins when text rendering, identity preservation, and prompt adherence matter.** Flux.1 [pro] sits second on the Artificial Analysis typography benchmark (~1068 ELO, behind Ideogram 3) and renders correct text where SDXL reliably hallucinates characters [6b]. But Flux tilts photographic, so flat-vector icon LoRAs are often easier to land on SDXL [6d].
5. **SDXL's dual encoder is a tool, not a redundancy.** Splitting `text_g` (OpenCLIP-bigG, 1280-dim) for composition/mood and `text_l` (CLIP-L, 768-dim) for colors, materials, named entities yields measurably better SDXL output than sending the same prompt twice [6a]. A1111 abstracts this away; ComfyUI's `CLIPTextEncodeSDXL` and micro-conditioning inputs (`original_size`, `crops_coords_top_left`, `target_size`) expose it [6a].
6. **`BREAK` is the single most under-used tool for asset prompting.** Because each 75-token CLIP chunk is encoded in isolation, putting subject tokens before `BREAK` and background tokens after prevents "indigo-to-violet gradient" from leaking onto the background you asked to keep white — the exact bug that wrecks logo/icon jobs on SDXL [6a]. Civitai's top logo prompts all use `BREAK`.
7. **Guidance-distilled models pay for faster inference with fragile artifact recipes.** The `"blurry, low-quality, watermark"` SDXL ritual has no direct translation to Flux. The working substitutes are (a) affirmative rewrites ("tack-sharp, clean un-watermarked composition, anatomically correct hands"), (b) `prompt_upsampling=true` on Flux.2 to pre-enrich terse inputs, and (c) routing edits through Kontext instead of re-rolling [6b].
8. **ControlNet + IP-Adapter is the canonical two-adapter stack for brand-consistent asset families.** Canny (weight 0.6) locks silhouette while IP-Adapter-Plus (weight 0.4) carries color/material language — the single most reliable open-source recipe for "16 sibling icons in one brand system from one logo" across SDXL, Flux.1, and Flux.2 [6c]. Stacking more than two adapters usually degrades quality.
9. **First-party Flux Tools dominate community adapters where they overlap.** Flux.1 Fill / Canny / Depth / Redux (Nov 2024) ship as BFL-trained guidance-distilled checkpoints and outperform community InstantX / XLabs adapters on their native modes; use InstantX/Shakker ControlNet-Union only for modes BFL doesn't ship (pose) [6c]. Kontext further collapses the edit loop — "logo → favicon → OG card → splash" becomes one conversation, not a re-roll-and-mask cycle [6c].
10. **PuLID is strictly better than InstantID for human identity on SDXL; PuLID-Flux is the Flux default.** PuLID's contrastive alignment keeps non-ID attributes (pose, style, background) close to the base model's un-identity-conditioned output, so style degradation is dramatically lower than InstantID [6c]. For non-human mascots, IP-Adapter-Plus at 0.7–0.9 + optional ControlNet-OpenPose is the equivalent recipe [6c].
11. **LoRA is the lowest-risk, highest-leverage brand-consistency primitive.** 10–30 curated images, 1–4 hours on a 24 GB GPU (or $2–6 on Replicate/fal), rank 8 for flat icons / rank 16 with DoRA for painterly / rank 4 LoHA for logo marks — ships as a 20–400 MB adapter that slots into any SDXL or Flux pipeline [6d]. Flux + ostris `ai-toolkit` at rank 16–32 on 15–25 captioned images with a rare trigger token is the 2026 default.
12. **DoRA is a free quality upgrade over vanilla LoRA for style work.** Weight-Decomposed Low-Rank Adaptation (Liu et al. 2024) splits weights into magnitude and direction and applies the low-rank update only to the direction; at the same rank it gives sharper color fidelity and less style-leak into unrelated concepts, at ~2× training cost and zero inference cost [6d]. LoKr is the pick for <10 MB edge-shippable artifacts; LoHA helps on 5–10-image micro-datasets.
13. **Dataset and caption discipline dominate hyperparameter choice.** Rank 16 / alpha 16 is a safe LoRA default regardless of target; pushing rank to 64 won't rescue a noisy 200-image set or captions that leak the style into prose [6d]. The rule is: caption the *content*, let the rare trigger token own the style, sprinkle 5–10% caption dropout.
14. **ComfyUI behind `POST /prompt` is the production default for a multi-asset pipeline; Diffusers is the default when you need single-box throughput.** ComfyUI's typed node graph gives day-of access to every new adapter (Flux Tools, Kontext, LayerDiffuse, IP-Adapter-Plus) without code changes; Diffusers with `torch.compile` + `enable_model_cpu_offload` + bnb-4bit gives the tightest in-process control for 64-image batches; SwarmUI is the only open-source way to fan a single request across N backends cleanly [6e].
15. **Distilled-model pitfalls are real and concentrate in three places.** (a) Flux.1 [schnell]'s ADD step-distillation to 1–4 steps means `guidance_scale` is inert; (b) Flux.1 [dev]'s guidance distillation means no negatives and values above ~6 saturate / break hands; (c) stacking two strong adapters on distilled models forces weight reductions (−0.1 to −0.2 vs. SDXL equivalents) because the single forward pass has less headroom to absorb conflicting signals [6b][6c].

## Cross-Cutting Patterns

- **Two architectures, one enhancer contract.** SD (UNet + CLIP) and Flux (MM-DiT + T5/VLM) require different surface prompts but the enhancer's job is the same: produce subject/background separation, lock palette, normalize negatives, pick adapters. 6a, 6b, and 6d all converge on the idea that the enhancer should branch on `{family: "sd" | "flux"}` once and then apply model-family-appropriate rewrites downstream.
- **Affirmative descriptions beat prohibitions on both families.** On Flux this is forced (no negative prompt); on SDXL this is still better practice because negative prompts consume cross-attention bandwidth and A1111's renormalization reallocates it away from positives. 6a and 6b agree: "clean un-watermarked composition" beats `"watermark, text, signature"`.
- **Subject/background separation via explicit tokens.** SDXL uses `BREAK` [6a]; Flux uses prose sentence boundaries ("on a pure white background, centered with strong negative space") [6b]; both rely on the same mechanism — forcing the text encoder to segregate subject tokens from background tokens so that cross-attention spatial regions attend cleanly.
- **Silhouette + style is the durable two-channel composition.** Canny ControlNet (shape) + IP-Adapter-Plus (style) on SDXL [6c]; Flux Canny + Redux on Flux [6c]; ControlNet + LoRA trigger token [6c+6d]. The pattern generalizes: one conditioning signal for geometry, one for appearance.
- **1024×1024 is the modern native.** SDXL multi-aspect bucketing is weakest at sub-native [6a]; Flux.1 [dev] and Flux.2 both target 1024² or higher natively [6b]; LoRA training uniformly uses 1024² [6d]; ComfyUI/Diffusers batch recipes default to 1024² [6e]. Anything below loses detail to bucketing and VAE decode artifacts.
- **No model natively outputs reliable RGBA.** Every angle hands off transparency to a matting post-process: SDXL via `LayerDiffuse`, Flux via RMBG-1.4 / BiRefNet, and ComfyUI's LayerDiffuse node is the cleanest integrated path [6b][6d][6e]. See category 13 for the deep-dive.
- **LoRA + adapter stacking requires weight coordination.** `<lora:brand:0.9>` + IP-Adapter 0.5 on SDXL works; the Flux equivalent needs both weights reduced ~0.1–0.2 because of the distilled single forward pass having less headroom [6c][6d].

## Controversies / Disagreements

- **"Flux can do negative prompts with a workaround" vs. "No, it really can't."** BFL's Flux.2 prompting guide [6b ref 10] is unambiguous: no negative prompts. But community work — Forge's "Flux1D Distillation CFG" experiment and semi-de-distillation via inverse-cosine timestep remapping [6b ref 18] — shows negatives can be partially re-enabled by forcing a second forward pass and perturbing the schedule. 6b treats these as research curiosities, not production features, because no commercial endpoint exposes them and they cost 2× inference time. 6d notes in passing that "Flux ignores CFG-style negatives … though flux-dev does honor a weak negative via guidance_scale tricks," which is the softer framing. The consensus is: treat negatives as not-working for product code, surface the affirmative-rewrite table to users, and leave the hack to power users.
- **ComfyUI-Union ControlNet vs. specialized per-mode ControlNets.** 6c explicitly flags that "Union models are weaker per-mode" and that Jasperai's single-mode Flux Canny often beats InstantX Union on canny. But Union is cheaper to load/swap in pipelines. The recommendation splits: use Union for dev iteration, swap in a specialized for production quality if the pipeline is fixed to one mode.
- **LoRA rank for style: "start at 16" vs. "start at 8 for icons."** 6d recommends rank 16 as the universal safe default but rank 8 specifically for flat-vector icon packs (Recipe A), because higher rank overfits and memorizes specific icons rather than the style. This is the one place where the "just use 16" heuristic is wrong.
- **A1111 vs. ComfyUI weight semantics.** 6a documents that A1111 normalizes token weights across the prompt while ComfyUI does not, so identical prompts + seeds produce different images [6a ref 9]. The consensus fix is `ComfyUI_ADV_CLIP_emb` with `token_normalization=mean`, but this is not the ComfyUI default, so any cross-UI prompt-library must carry normalization metadata, not just the prompt string.
- **"Just add more images" vs. dataset quality for brand LoRAs.** 6d's headline heuristic — "a curated 15-image set beats a 200-image noisy set at any rank" — pushes back against Civitai community practice. Disagreement is about regularization: SDXL benefits from 50–200 regularization images; Flux community says skip them. Both are true, but only if dataset curation is actually done.
- **SD 1.5 is dead vs. SD 1.5 is still the right tool for flat icons.** 6a and 6d both note SD 1.5's continued utility for anime/illustration/flat-vector checkpoints where SDXL over-indexes on photoreal and Flux tilts further. No modern SOTA benchmark reports SD 1.5 favorably, but the community LoRA ecosystem is richest there.

## Gaps

- **No dedicated treatment of inpainting/outpainting prompt patterns within this category.** 6c covers Flux.1 Fill at the adapter level but prompt-side masking ergonomics (mask-aware prompt design, region-specific prompts via Regional Prompter or ComfyUI Impact-Pack) is not deep-dived. Cross-reference category 16 and category 18.
- **Aspect-ratio and multi-scale guidance for OG/Twitter/favicon targets.** The angles mention 1024² but platform-specific ratios (1200×630 OG, 1080×1080 Instagram, 180×180 Apple touch icon) are handled in category 11 / 18. The enhancer should own the mapping; this category doesn't.
- **VAE choice and color-accuracy impact on brand palette fidelity.** 6a documents VAE as "the pixel-space autoencoder" but doesn't cover the SDXL VAE 0.9 vs. 1.0 vs. madebyollin SDXL-FP16 VAE choice, which materially affects color accuracy for brand work.
- **Serving economics.** 6e gives VRAM profile and cloud options but not a full cost-per-1000-assets comparison across Replicate vs. fal vs. self-hosted A100 vs. RunPod spot. The plugin roadmap needs this.
- **Quantitative benchmarks for text rendering inside logos/wordmarks.** 6b cites ELO and correctness ratios but doesn't include a reproducible test harness the plugin could use as CI.
- **Flux.2 ecosystem coverage.** Flux.2 launched 2025-11-25 (inside the research window) but ComfyUI / Forge / Diffusers custom-node maturity is still settling; 6e's matrix assumes Flux.1 for most rows.
- **Agentic use of Kontext as an iterative edit tool.** 6b flags Kontext as "the primitive that makes iterative asset generation economically viable" but there's no worked example of a multi-turn agentic loop (generate → critique → Kontext-edit → accept). This connects to category 19.
- **Legal/licensing coverage.** FLUX-1 Non-Commercial, FLUX-2 Non-Commercial, Apache 2.0 (schnell, klein, VAE) matter for the plugin's commercial posture; 6b lists licenses but doesn't translate them into product decisions.

## Actionable Recommendations for the Plugin

### Self-hostable pipeline recommendation

- **Default runtime: ComfyUI behind `POST /prompt`** [6e], running on a 24 GB GPU (4090 / A5000 / L40S) or the `runpod/worker-comfyui` serverless image for bursty workloads. Ship a versioned library of API-format workflow JSON templates per asset category (logo, app icon, illustration, OG card, favicon, splash). Parametrize by (a) prompt text node, (b) seed, (c) filename_prefix, (d) checkpoint path, (e) LoRA stack, (f) adapter weights. Install the must-have custom-node set documented in 6e: `ComfyUI-Manager`, `rgthree-comfy`, `ComfyUI-Impact-Pack`, `was-node-suite`, `ComfyUI-GGUF`, `ComfyUI-IPAdapter-plus`, `ComfyUI-ControlNet-Aux`, `ComfyUI-LayerDiffuse`.
- **Low-VRAM path: Forge with NF4 Flux + A1111-shape API** [6e]. Any existing `sdwebuiapi` client works unchanged. Useful for enthusiast tier / user-owned GPU (12–16 GB).
- **High-throughput path: Diffusers in-process with `torch.compile` + `enable_model_cpu_offload` + bnb-4bit** [6e]. 64-image batches via `itertools.product` over (subject × seed × style × aspect) with inner `BATCH=4`. Wrap in `asyncio.Queue` for request-level concurrency.
- **Multi-box path: SwarmUI fronting N ComfyUI backends** [6e]. One session per plugin instance; `GenerateText2ImageWS` fans the request across backends automatically.
- **Model defaults**: Flux.1 [dev] for typography / identity / prompt-adherence jobs; Flux.1 [schnell] for fast iteration; SDXL (Juggernaut XL / RealVis XL for photoreal, Vector illustration SDXL LoRA stack for flat assets) for flat/vector icon work and where VRAM is tight; Flux.2 [dev]/[flex] once the ecosystem stabilizes and for any job that needs JSON-hex-color prompts or multi-reference.

### ControlNet / IP-Adapter strategy

- **Expose an "Asset consistency mode"** [6c] that toggles between four pre-wired adapter stacks: (1) **Silhouette-locked icon family**: Canny (0.6) + IP-Adapter-Plus (0.4) from a single reference logo. (2) **Logo stamping**: scene without logo → Flux Fill with rectangular mask + XLabs Flux IP-Adapter v2 (0.9) pointing at the logo asset. (3) **Shape-locked color variants**: Flux Canny (0.8) from binary silhouette mask + prompt-varied material/color. (4) **Brand palette transfer**: Flux Redux (0.7–1.0) from a 4–6-image style sheet, optionally stacked with XLabs Flux IP-Adapter v2 at 0.3.
- **Cap stacks at two adapters** [6c]. Surface a UI warning if a user tries to add a third; internally enforce weight re-normalization to keep sum ≤ 1.3.
- **Prefer first-party Flux Tools** [6c]: BFL Canny / Depth / Fill / Redux over community InstantX/XLabs for their native modes. Fall back to InstantX Union / Shakker Union only for modes BFL doesn't ship (pose).
- **Never use the base model to render logo text** [6c]. Generate the scene, composite the vector logo via image-editor post-process, or use Flux Fill with IP-Adapter pointed at the clean logo PNG.
- **Human identity: PuLID-Flux by default** [6c]; InstantID as SDXL fallback. Hard-disable identity adapters unless the user has uploaded a consented reference (regulatory posture for EU AI Act + US state laws).

### LoRA training hook

- **Ship a "Train a brand LoRA" flow** [6d] that accepts 10–30 images and runs one of three presets: (A) flat-vector icon pack (rank 8, alpha 8, 2k steps, Flux ostris `ai-toolkit` or SDXL kohya), (B) painterly marketing illustration (rank 16 with DoRA, 4k steps), (C) duotone brand (rank 8–16 with hex-anchored captions), (D) logo mark (LoHA rank 4 with 50–100 regularization images).
- **Backend**: Replicate `ostris/flux-dev-lora-trainer` or fal.ai `flux-lora-fast-training` for $2–6 per brand, with an optional RunPod-H100 + ai-toolkit queue path for self-hosted.
- **Captioning pipeline**: auto-caption with BLIP-2 / CogVLM → strip style words → inject rare trigger token as first word → expose the captions for user review before training kicks off. This is the single biggest quality lever [6d].
- **Inference integration**: the enhancer must know (a) the trigger token for each brand LoRA, (b) the LoRA weight range (default 0.8–1.0 Flux, 0.65–0.9 SDXL), (c) the asset-template whitelist (flat icon vs. illustration) the LoRA was trained for. Store these as metadata alongside the safetensors file.
- **Multi-brand serving**: keep the base model shared; hot-swap adapters via `diffusers.set_adapters()` or ComfyUI Power Lora Loader (rgthree). Don't merge-and-unload per-brand unless the brand is so high-volume it warrants its own dedicated endpoint — loss of dynamic scale control outweighs the ~2% inference speedup.
- **DoRA by default for painterly styles; LoKr when artifact size matters for edge/CDN** [6d].

### Prompt-enhancer contract (family-aware)

- Detect `{family: "sd" | "flux"}` from the configured pipeline. For **SD**: emit A1111-DSL with `BREAK` subject/background separation, weighted quality-tag header for illustration checkpoints, `clip_skip=2` for NAI-lineage anime/flat, `(masterpiece:1.2), (best quality:1.3)` prefix only on checkpoints that re-learned it. Split SDXL prompts into `text_g` (composition) and `text_l` (attributes) when the backend supports it. For **Flux**: emit Subject+Action+Style+Context prose, convert negatives to affirmatives, default `guidance_scale=3.5`, set `max_sequence_length=512`; on Flux.2 emit JSON with `color_palette` hex codes and offer `prompt_upsampling=true` for short inputs.
